react的基础使用

1. React简介

1.官网

  1. 英文官网: https://reactjs.org/
  2. 中文官网: https://react.docschina.org/

2.介绍描述

我们下载下来可以看到这样一些文件

  1. react.js：React核心库。
  2. react-dom.js：提供操作DOM的react扩展库。
  3. babel.min.js：解析JSX语法代码转为JS代码的库。

react是使用jsx的，我们浏览器只能解析js，所以必须把jsx->js

2.hello_react

使用react分3步：

  1. 准备好一个“容器”
  2. 创建虚拟DOM
  3. 渲染虚拟DOM到页面

3.jsx

1.为什么要用jsx

  1. 全称: JavaScript XML
  2. react定义的一种类似于XML的JS扩展语法: JS + XML本质是React.createElement(component,props, ...children)方法的语法糖
  3. 作用: 用来简化创建虚拟DOM

我们的react就是创建虚拟DOM在把他转换为真实的DOM，用js的话需要用React提供的API，不好写

2.jsx语法规则

1.定义虚拟DOM时，不要写引号。

2.标签中混入JS表达式时要用{}。

3.样式的类名指定不要用class，要用className。

4.内联样式，要用style={{key:value}}的形式去写。

5.只有一个根标签

6.标签必须闭合

7.标签首字母

(1).若小写字母开头，则将该标签转为html中同名元素，若html中无该标签对应的同名元素，则报错。

就是去找html里的找不到报错

(2).若大写字母开头，react就去渲染对应的组件，若组件没有定义，则报错。

找组件里面的，找不到报错

一个小练习：

要动态更新数据就要{}，但是{}里面只能写js表达式

1.表达式：一个表达式会产生一个值，可以放在任何一个需要值的地方

用Const x = 来接，接得到就是

(1). a

(2). a+b

(3). demo(1)

(4). arr.map()

(5). function test () {}

2.语句(代码)：

(1).if(){}

(2).for(){}

3).switch(){case:xxxx}

2.React面向组件编程

1.定义组件

1.函数式组件

注意：

  • 函数里没有this
  • 我们没有自己调用函数，react会调用该函数，将返回的虚拟DOM转为真实DOM，随后呈现在页面中。

2.类式组件

注意：

声明的类要继承React.Component

我们的主要结构要放到render(){表达式}里面去

render里面的this指向MyComponent的实例对象

2.组件实例的三大属性

这三大属性都是针对类式组件的，因为函数组件没有实例，没有this。

1.state

首先，我们要创建一个类。

类里面三个重要的部分：

1.构造器：

  • 在普通的构造器里，我们想要继承父类，需要带父类的参数，再super()这些，而在react，父类参数用props代替
  • 我们要添加一个state，通常是个对象
  • 如果我们想在下面定义一个函数，很显然，普通函数this为undefined，于是我们在来添加一个属性，让这个在原型对象上的函数的this指向实例对象，我们就可以在reader里面通过this.函数名直接拿到这个函数了。

1.state简写方式

注意：

  • 我们为什么要constructor，是要新添一个属性，而且可能有传过来的参数constructor(props，name…),但我们只想新增一个参数固定的state，相当于新赋一个值，我们可以直接用对象法。
  • 普通函数的this指向function，也就是window，但是箭头函数的this是他的上一层函数的this，他的上一层是什么weather的实例对象

2.props

props有什么用？

在state里，我们知道state是用来存我们的数据的，但这个state是在的类里面的，数据也是放在类里面的。

如果我们想从外面传数据到类里面，我们就要用props

我们可以看到，const {name,age,sex} = this.props将数据从外面传进来

注意：

  1. age={19}这是把age用number类型传进来，不然是string类型
  2. 我们一般需要规定别人传进来的类型
  3. <Person {...p}/>，…p 是babel加react可以展开一个对象，只能适用标签属性。
  4. 其他时候，…p不能展开对象，当然js里可以用字面量复制对象

1.对props进行限制

刚才我们说了，一般要对props传进来的数据类型做限制

注意：

propTypes是一个规矩，而PropTypes是一个属性，他不是react自带的，要导入

2.props的简写方式

我们的Person.propTypes和Person.defaultProps在person类的外面，这样不规范，应该把他们放在类里

3.函数组件使用props

函数组件因为没有实例对象，没有this，无法使用state和ref，但他可以使用props，因为函数可以自带参数

3.refs

我们在js里，要拿到一个元素再操作它，我们要用到document.querySelector，react里我们用ref属性里代替它

refs相当于一个标签，组件内的标签可以定义ref属性来标识自己。

1.字符串形式的ref

字符串形式ref基本形式为：

  1. 实例对象上有一个refs属性
  2. 在一个元素里添加，ref="input1"，另一个添加，ref="input2"
  3. refs: {input1: input, input2: input}
  4. 我们就拿到了这两个input元素，就可以操作元素了

4.函数柯里化

刚才的两个案例，我们注意到我们写了两个函数来更新state里的username和password，这样代码复用性不好

我们想能不能把写一个函数代替这两个函数，这就引出了函数柯里化

函数的柯里化：

通过函数调用继续返回函数的方式，实现多次接收参数最后统一处理的函数编码形式。

我们看看改后的代码：

saveFormData 代替了这两个

  • 刚才onChange里的是this.saveUsername，而saveUsername是一个回调函数，我们触发的是一个函数
  • onChange里的是this.saveFormData('username')，而saveFormData('username')是一个返回值，我们触发的是undefined，所以我们要返回一个回调函数。
  • 而this.state[dataType]是一种对象的写法，是让dataType的值填进去

我们也可以不用柯里化来写

onChange={event => this.saveFormData('username',event) } 直接在这里面写回调函数，我们返回的还是一个函数

5.组件生命周期🔥

这个是组件的生命周期，就是写在类里面的。

1.react生命周期(旧)

什么是挂载？

我们把React.js将组件渲染，并且构造 DOM 元素然后塞入页面的过程称为组件的挂载

我们分两个阶段来记：

  1. 初始化阶段: 由ReactDOM.render()触发—初次渲染
名称 作用时间
constructor() 最开始
componentWillMount()(了解) 组件将要渲染
render()
componentDidMount()🔥 组件被渲染好了

componentDidMount() =====> 常用

一般在这个钩子中做一些初始化的事，例如：开启定时器、发送网络请求、订阅消息

  1. 更新阶段: 由组件内部this.setSate()或父组件render触发
名称 作用时间
shouldComponentUpdate()🔥 组件更新的阀门
componentWillUpdate() 组件将要更新
render() =====> 必须使用的一个
componentDidUpdate()

shouldComponentUpdate()🔥

默认属性是true，如果写出这个要加上return true还是false，判断可不可以更新

  1. 卸载组件: 由ReactDOM.unmountComponentAtNode()触发

componentWillUnmount() =====> 常用

一般在这个钩子中做一些收尾的事，例如：关闭定时器、取消订阅消息

2.react生命周期(新)

componentWillMount()

componentWillUpdate()

componentWillReceiveProps()

被取消，第一个新增的是

getDerivedStateFromProps ，这个没用什么用

getSnapshotBeforeUpdate()，这个可以拿到更新之前的数据，Snapshot是快照的意思，把更新前的拍下来

5.Dom的diff算法

这个东西，一般是在遍历的时候使用。

我们习惯用index，做为li的key值，这样会有很大的隐患

为什么？

虚拟DOM中key的作用：

简单的说: key是虚拟DOM对象的标识。

详细的说: 当状态中的数据发生变化时，react会根据【新数据】生成【新的虚拟DOM】, 随后React进行【新虚拟DOM】与【旧虚拟DOM】的diff比较，比较规则如下：

a. 旧虚拟DOM中找到了与新虚拟DOM相同的key：

​ (1).若虚拟DOM中内容没变, 直接使用之前的真实DOM

​ (2).若虚拟DOM中内容变了, 则生成新的真实DOM，随后替换掉页面中之前的真实DOM

b. 旧虚拟DOM中未找到与新虚拟DOM相同的key

根据数据创建新的真实DOM，随后渲染到到页面

看一个例子：

初始数据：

{id:1,name:'小张',age:18},

{id:2,name:'小李',age:19},

初始的虚拟DOM：

<li key=0>小张---18<input type="text"/></li>

<li key=1>小李---19<input type="text"/></li>

更新后的数据：

{id:3,name:'小王',age:20},

{id:1,name:'小张',age:18},

{id:2,name:'小李',age:19},

更新数据后的虚拟DOM：

<li key=0>小王---20<input type="text"/></li>

<li key=1>小张---18<input type="text"/></li>

<li key=2>小李---19<input type="text"/></li>

  1. 我们更新后，所有的数据key都改变，我们需要都更新Dom，这样影响效率
  2. 输入类的DOM，我们输进去的东西会留在input里，对比两个dom，我们发现<input type="text"/>是相同的，于是，就直接渲染来用

用index作为key可能会引发的问题：

  1. 若对数据进行：逆序添加、逆序删除等破坏顺序操作:

会产生没有必要的真实DOM更新 ==> 界面效果没问题, 但效率低。

  1. 如果结构中还包含输入类的DOM：

会产生错误DOM更新 ==> 界面有问题。

  1. 注意！如果不存在对数据的逆序添加、逆序删除等破坏顺序操作，

仅用于渲染列表用于展示，使用index作为key是没有问题的。

开发中如何选择key?:

1.最好使用每条数据的唯一标识作为key, 比如id、手机号、身份证号、学号等唯一值。

2.如果确定只是简单的展示数据，用index也是可以的。

2.用index作为key可能会引发的问题：

若对数据进行：逆序添加、逆序删除等破坏顺序操作:

会产生没有必要的真实DOM更新 ==> 界面效果没问题, 但效率低。

如果结构中还包含输入类的DOM：

会产生错误DOM更新 ==> 界面有问题。

注意！如果不存在对数据的逆序添加、逆序删除等破坏顺序操作，

仅用于渲染列表用于展示，使用index作为key是没有问题的。
